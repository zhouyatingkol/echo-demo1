# ECHO 核心机制：权属蓝图与关系图谱验证
**验证的不是"使用"，是"关系是否合法"**

---

## 一、核心洞察

### 1.1 重新定义 ECHO

**不是：**
- 记录使用数据的区块链
- 证明作品存在的存证系统
- 追踪版权的登记平台

**而是：**
- **验证关系合法性的协议**
- 检查"实际发生的关系"是否符合"预设的规则蓝图"
- 带着关系的资产，在关系中生长

### 1.2 两个核心概念

```
权属蓝图（静态规则）          关系图谱（动态事实）
    │                              │
    ├── 四权力配置                 ├── 使用关系
    │   ├── 用：谁能接触           │   └── 谁在何时用了作品
    │   ├── 扩：谁能传播           │
    │   ├── 衍：谁能改编           ├── 衍生关系
    │   └── 益：收益怎么分         │   └── 谁基于谁创作了
    │                              │
    ├── 治理模式                   ├── 引用关系
    │   └── 规则能不能改           │   └── 谁引用了谁，比例多少
    │                              │
    ├── 衍生费比例                 ├── 交易关系
    │   └── 用了要付多少           │   └── 谁买了谁的收益权
    │                              │
    └── 创作手记                   └── 变更关系
        └── 意图说明                   └── 规则何时如何变更

验证的核心问题：
"实际发生的关系" 是否符合 "权属蓝图预设的规则"？
```

## 二、验证什么？

### 2.1 使用关系验证

```
实际发生：小明访问了《墨竹》
    ↓
验证内容：
├── 《墨竹》的"用"档位是否允许小明访问？
├── 访问次数是否超过限制？
├── 访问平台是否符合"扩"的范围？
└── 是否需要付费（根据"益"的设定）？

验证结果：
✅ 合法：记录关系，更新六爻
❌ 非法：拒绝访问，记录违约
```

### 2.2 衍生关系验证

```
实际发生：小刚基于《墨竹》创作了《融》
    ↓
验证内容：
├── 《墨竹》的"衍"档位是否允许混剪/改编？
├── 小刚是否获得父代（小明）的确认？
├── 衍生费比例之和是否 ≤ 100%？
│   ├── 《墨竹》：30%（小明设定）
│   ├── 《山水》：40%（小红设定）
│   └── 《诗词》：30%（小绿设定）
│   └── 总计：100% ✅
└── 新作品的治理模式是否符合父代预期？

验证结果：
✅ 合法：创建《融》，锁定父代权益，分配收益权
❌ 非法：拒绝创建，或要求调整配置
```

### 2.3 引用关系验证

```
实际发生：小美引用了《融》，而《融》引用了《墨竹》
    ↓
验证内容：
├── 引用链是否完整？（《墨竹》→《融》→小美作品）
├── 每层引用是否符合父代的衍生费设定？
├── 收益分配是否自动执行？
│   └── 小美交易 → 分给《融》→ 分给《墨竹》
└── Attribution 链是否正确显示？

验证结果：
✅ 合法：自动执行收益分配，更新引用图谱
❌ 非法：标记为侵权，启动纠纷处理
```

### 2.4 交易关系验证

```
实际发生：小红购买了《墨竹》30%收益权
    ↓
验证内容：
├── 出售者是否有权出售？（持有收益权）
├── 交易是否符合"益"的档位限制？
│   └── 益=2-稀（限量99份），当前已售多少份？
├── 购买者是否符合使用权限？（根据"用"的设定）
└── 交易后收益分配比例是否正确更新？

验证结果：
✅ 合法：转移收益权，更新持有者列表
❌ 非法：拒绝交易，说明原因
```

### 2.5 变更关系验证

```
实际发生：小明想修改《墨竹》的四权力配置
    ↓
验证内容：
├── 当前治理模式是否允许变更？
│   ├── 永恒承诺：❌ 不允许
│   ├── 有限治理：✅ 允许，但需满足条件
│   └── 灵活治理：✅ 允许
├── 是否提前90天公告？（有限治理要求）
├── 旧版本持有者是否获得原价退出权？
└── 变更是否影响已存在的衍生作品？
    └── 冻结保护：已授权部分不可改

验证结果：
✅ 合法：发布新版本，旧版本继续有效
❌ 非法：拒绝变更，或要求补充条件
```

## 三、链上存储的内容

### 3.1 不是"数据"，是"合法性证明"

**传统区块链存证：**
```
区块记录：
{
  "timestamp": "2026-03-20T20:00:00Z",
  "event": "小明访问了《墨竹》",
  "hash": "0x..."
}
```

**ECHO 链上记录：**
```
区块记录：
{
  "timestamp": "2026-03-20T20:00:00Z",
  "event": "使用关系",
  "subject": "小明",
  "object": "《墨竹》",
  "blueprintCheck": {
    "yong": "5-用",
    "allowed": true,
    "limit": "100次/天",
    "current": 23
  },
  "valid": true,
  "hash": "0x..."
}
```

**关键区别：**
- 不是"发生了什么"，是"发生了什么，且符合规则"

### 3.2 存储结构

```
链上轻量存储（验证结果）：
├── 关系类型哈希
├── 合法性布尔值
├── 规则版本号
└── 时间戳

链下完整存储（关系细节）：
├── 完整关系数据
├── 三维编织结构
├── 六爻状态
└── 引用图谱
```

## 四、验证机制的技术实现

### 4.1 验证流程

```
关系事件提交
    ↓
调取权属蓝图（静态规则）
    ↓
检查关系是否符合规则
    ↓
├── 符合 → 记录"合法关系"，更新图谱
└── 不符合 → 记录"非法关系"，拒绝/告警
```

### 4.2 验证者角色

| 验证者 | 职责 | 激励 |
|:-------|:-----|:-----|
| **创作者节点** | 验证是否符合自己的蓝图 | 保护自身权益 |
| **平台节点** | 验证使用行为真实性 | 平台声誉 |
| **社区节点** | 抽查验证，举报异常 | 奖励机制 |
| **智能合约** | 自动验证规则一致性 | 无需激励 |

### 4.3 争议处理

```
争议提交："我认为这次使用不合法"
    ↓
调取链上记录：当时的蓝图版本 + 关系数据
    ↓
社区仲裁（如果涉及复杂解释）
    ↓
├── 维持原判：记录仲裁结果，惩罚申诉方
└── 推翻原判：回滚关系，补偿受损方
```

## 五、一句话总结

> **ECHO 链上记录的，不是"发生了什么"，是"发生了什么，且符合预设的规则"。**

**带着关系的资产：**
- 资产不是孤立的存在
- 是它所处关系的节点
- 每个关系都被验证是否符合创作者的意图

**这就是 ECHO 解决权利冲突的方式：**
- 不是事后扯皮
- 是事前设定规则
- 是实时验证关系
- 是自动执行后果
