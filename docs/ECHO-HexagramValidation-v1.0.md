## 一、核心设计

### 1.1 64卦作为验证层

**不是：**
- 64卦只是状态描述（之前的设计）
- 四权力完全由创作者自由设定
- 验证只检查链上签名

**而是：**
- **64卦是"物理法则"**，限制四权力的可行范围
- **四权力是"初始条件"**，必须在法则允许内设定
- **验证首先检查**：这个配置在当前生命状态下是否可能

### 1.2 类比理解

```
物理世界：
├── 物体有质量（属性）
├── 质量产生引力（法则）
└── 你不能让石头向上飞（违背法则）

ECHO世界：
├── 资产有生命状态（卦象）
├── 状态产生约束（法则）
└── 你不能让潜藏资产全开放（违背法则）
```

**这不是平台限制，是自然法则。**

## 三、验证流程

### 3.1 标准验证流程

```
输入：
├── 当前卦象（如：初九）
├── 当前六爻坐标（如：T[0][0][0]）
└── 提议的四权力配置（如：{用:7, 扩:5, 衍:4, 益:2}）

验证步骤：
1. 查询验证规则表
   └── 初九规则：max_yong=3, max_kuo=2, max_yan=2

2. 逐项检查
   ├── 用=7 > max_yong=3？ ❌ 非法
   ├── 扩=5 > max_kuo=2？ ❌ 非法
   ├── 衍=4 > max_yan=2？ ❌ 非法
   └── 益=2：无约束，✓

3. 综合判断
   └── 存在非法项，拒绝配置

输出：
❌ 验证失败
📋 原因："当前状态为初九·潜藏，不可设置如此开放的权限"
💡 建议："用≤3, 扩≤2, 衍≤2，或先让资产生长到九二"
```

### 3.2 不同触发时机的验证

#### 时机1：铸造时

```
输入：初始四权力配置
验证：
├── 新资产通常在坤卦或乾卦初始状态
├── 检查是否符合初始卦象约束
└── 如果不符合，提示调整

示例：
用户设置：用=7, 扩=7, 衍=7, 益=0（全开放，无收益限制）
初始卦象：坤卦（全低）
验证结果：❌ 坤卦不可如此开放
建议："新资产建议从适度开放开始，随生长逐步扩展"
```

#### 时机2：四权力变更时

```
输入：新的四权力配置
验证：
├── 查询当前卦象
├── 检查新配置是否符合当前生命状态
└── 如果不符合，拒绝变更

示例：
当前卦象：九五·飞龙在天（T[2][2][2]）
用户想改：用=1（锁定），扩=0（独享），衍=0（禁止）
验证结果：❌ 大成之物不可全封闭
原因："已有5个衍生作品，30个引用，物理上无法封闭"
建议："用≥4, 扩≥3, 衍≥3，或等待上九转化期"
```

#### 时机3：变卦时（六爻变化）

```
触发：资产从九四变到九五
验证：
├── 新卦象九五的约束：用≥4, 扩≥3, 衍≥3
├── 检查当前四权力是否符合新约束
└── 如果不符合，强制调整或阻止变卦

示例：
变卦前：九四，配置 {用:3, 扩:3, 衍:3, 益:4}
变卦后：九五，约束 {min_yong:4, min_kuo:3, min_yan:3}
检查：
├── 用=3 < min_yong=4？❌ 不符合
└── 处理：
    ├── 选项A：强制调整用=4（自动升级）
    ├── 选项B：阻止变卦，保持九四（人工选择）
    └── 选项C：提示创作者必须调整（推荐）
```

#### 时机4：治理模式变更时

```
触发：用户想从"有限治理"改为"永恒承诺"
验证：
├── 检查当前卦象是否允许改为永恒
├── 九五/上九状态：❌ 不可（已有生态）
├── 初九/九二状态：✓ 可以（尚无依赖）
└── 乾卦/坤卦：✓ 可以（极端状态稳定）

示例：
当前：九五·飞龙在天，有限治理
用户想改：永恒承诺
验证结果：❌ 大成之物有衍生生态，不可改为永恒
原因："已有5个衍生者依赖当前规则，改为永恒将冻结权益"
建议："需获得所有衍生者同意，或等待上九转化期"
```

## 四、解决的权利冲突

### 4.1 创作者 vs 物理现实

```
冲突：创作者想让潜藏资产全开放（用=7）
现实：初九状态物理上不可能全开放
解决：
├── 验证拒绝："潜藏之物不可全开放"
├── 不是限制创作者，是保护物理一致性
└── 建议：先让资产生长，再逐步开放
```

### 4.2 创作者 vs 衍生者

```
冲突：大成资产的创作者想突然封闭（禁止衍生）
现实：已有5个衍生作品，物理上无法切断
解决：
├── 验证拒绝："大成之物不可全封闭"
├── 保护衍生者的既有权益
└── 建议：维持开放或协商退出
```

### 4.3 短期利益 vs 长期生长

```
冲突：创作者想高价限量（益=0）但全开放传播（扩=7）
现实：大量传播但限量，市场会混乱
解决：
├── 验证检查：扩=7时益不可=0
├── 高传播+高稀缺 = 不可持续
└── 建议：平衡配置，或调整卦象
```

### 6.1 验证合约

```solidity
contract HexagramValidation {
    
    struct ValidationRule {
        uint8 minYong;
        uint8 maxYong;
        uint8 minKuo;
        uint8 maxKuo;
        uint8 minYan;
        uint8 maxYan;
        bool governanceLock;
        string reason;
    }
    
    mapping(uint8 => ValidationRule) public rules; // hexagramID => rule
    
    function validate(
        uint8 currentHexagram,
        uint8[4] memory proposedRights // [yong, kuo, yan, yi]
    ) public view returns (bool isValid, string memory message) {
        
        ValidationRule memory rule = rules[currentHexagram];
        
        // 检查最小值
        if (proposedRights[0] < rule.minYong) {
            return (false, "用权低于当前生命状态允许的最小值");
        }
        // ... 其他检查
        
        return (true, "验证通过");
    }
}
```
