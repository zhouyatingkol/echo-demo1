# ECHO 64卦索引机制
**三维坐标映射 + 动态变卦算法**

---

## 一、核心设计

### 1.1 64卦作为生命状态索引

**三维坐标 → 6位二进制 → 64卦**

```
时间维度（2位）+ 空间维度（2位）+ 关系维度（2位）= 6爻
        ↓
    6位二进制码
        ↓
    查表得64卦之一
        ↓
    倒排索引快速定位
```

**数学基础：**
- 6爻 × 2种状态 = 64种组合
- 正好对应6位二进制数（0~63）
- 每卦代表一种独特的"生命状态"

### 1.2 三维映射规则

| 维度 | 位 | 状态定义 |
|:-----|:--:|:---------|
| **时间** | 5-4位 | 00=低活跃, 01=中活跃, 10=高活跃, 11=转化期 |
| **空间** | 3-2位 | 00=单平台, 01=少平台, 10=多平台, 11=跨域 |
| **关系** | 1-0位 | 00=孤立, 01=弱连接, 10=中连接, 11=强生态 |

**示例映射：**
```
时间=高活跃(10) + 空间=多平台(10) + 关系=强生态(11)
        ↓
    二进制：101011
        ↓
    十进制：43
        ↓
    卦象：泽山咸
```

## 二、数据结构

### 2.1 倒排索引

```json
{
  "hexagramIndex": {
    "1": ["assetA", "assetB", "assetC"],    // 乾卦
    "14": ["assetD", "assetE"],              // 火风鼎
    "43": ["assetF"],                        // 泽山咸
    "...": []
  }
}
```

**特点：**
- O(1) 查询某卦象的所有资产
- 动态更新（变卦时移动资产）
- 支持模糊查询（相似卦象）

### 2.2 资产状态

```json
{
  "assetState": {
    "assetA": {
      "currentHexagram": 14,           // 当前卦象
      "binaryState": "001110",         // 6位二进制
      "dimensions": {
        "time": "medium",              // 中活跃
        "space": "wide",               // 多平台
        "relation": "strong"           // 强连接
      },
      "trajectory": [1, 14, 50],       // 历史卦象变化
      "lastUpdated": "2026-03-20T20:00:00Z",
      "cooldownEnd": "2026-03-27T20:00:00Z"  // 冷却期结束
    }
  }
}
```

## 三、查询算法

### 3.1 精确查询

**查询某资产的当前状态**

```
输入：assetID
输出：currentHexagram, dimensions, binaryState

算法：
1. 查 assetState[assetID]
2. 返回完整状态信息

时间复杂度：O(1)
```

### 3.2 范围查询

**查询某卦象的所有资产**

```
输入：hexagramID (如 14)
输出：该卦象下的资产列表

算法：
1. 查 hexagramIndex[hexagramID]
2. 返回资产ID列表

时间复杂度：O(1) 获取列表，O(n) 遍历
```

### 3.3 模糊查询（相似状态）

**查询与当前资产状态相似的资产**

```
输入：currentHexagram (如 14=001110), similarityThreshold (如 1)
输出：相似卦象的资产列表

算法：
1. 当前卦象转二进制：14 → 001110
2. 计算汉明距离 ≤ threshold 的所有卦象：
   - 001111 = 15（火水未济）
   - 001010 = 10（天风姤）
   - 000110 = 6（天水讼）
   - 011110 = 30（离为火）
   - 101110 = 46（泽风大过）
3. 合并这些卦象的资产列表

时间复杂度：O(6) = O(1) 找相似，O(kn) 合并
```

### 3.4 轨迹查询

**查询经历过特定变卦轨迹的资产**

```
输入：startHexagram, endHexagram
输出：经历过该变化的资产列表

算法（优化后）：
1. 查变卦索引 transitionIndex[start][end]
2. 返回资产列表

时间复杂度：O(1)
```

**变卦索引结构：**
```json
{
  "transitionIndex": {
    "1": {
      "14": ["assetA", "assetB"],    // 乾→大有
      "2": ["assetC"]                 // 乾→坤
    },
    "14": {
      "50": ["assetD"]                // 大有→鼎
    }
  }
}
```

### 3.5 复合查询

**多维度条件查询**

```
输入：dimensionFilters = {time: "high", space: "wide", relation: "strong"}
输出：匹配资产列表

算法：
1. 维度值转二进制：
   time="high" → 10
   space="wide" → 10
   relation="strong" → 11
2. 组合：10 10 11 = 101011 = 43
3. 查 hexagramIndex[43]
4. 如有其他条件（如创建时间范围），再过滤

时间复杂度：O(1) + O(filter)
```

## 四、变卦触发机制

### 4.1 维度阈值定义

**时间维度（2位）：**

| 值 | 状态 | 触发条件 | 检查周期 |
|:--:|:-----|:---------|:--------:|
| 00 | 低活跃 | 7天内使用 < 3次 | 每日 |
| 01 | 中活跃 | 7天内使用 3-10次 | 每日 |
| 10 | 高活跃 | 7天内使用 > 10次 | 每日 |
| 11 | 转化期 | 30天无新使用 或 社区投票 | 事件触发 |

**空间维度（2位）：**

| 值 | 状态 | 触发条件 | 检查周期 |
|:--:|:-----|:---------|:--------:|
| 00 | 单平台 | 仅1个平台有记录 | 每周 |
| 01 | 少平台 | 2-3个平台 | 每周 |
| 10 | 多平台 | 4-7个平台 | 每周 |
| 11 | 跨域 | 8+平台 或 线上线下联动 | 每周 |

**关系维度（2位）：**

| 值 | 状态 | 触发条件 | 检查周期 |
|:--:|:-----|:---------|:--------:|
| 00 | 孤立 | 无引用、无衍生 | 实时 |
| 01 | 弱连接 | 被引用 1-5次 | 实时 |
| 10 | 中连接 | 被引用 6-20次 或 有衍生 | 实时 |
| 11 | 强生态 | 被引用 >20次 或 有二次衍生 | 实时 |

### 4.2 变卦算法

```python
def evaluate_hexagram(asset_id):
    """评估资产当前卦象"""
    
    # 获取数据
    usage_data = get_recent_usage(asset_id, days=7)
    platform_data = get_platform_distribution(asset_id)
    relation_data = get_citation_graph(asset_id)
    
    # 计算各维度值
    time_score = calculate_time_score(usage_data)      # 0-3
    space_score = calculate_space_score(platform_data)  # 0-3
    relation_score = calculate_relation_score(relation_data)  # 0-3
    
    # 组合二进制
    binary_state = f"{time_score:02b}{space_score:02b}{relation_score:02b}"
    new_hexagram = int(binary_state, 2)
    
    return new_hexagram, binary_state

def trigger_hexagram_change(asset_id):
    """触发变卦"""
    
    asset = assetState[asset_id]
    
    # 检查冷却期
    if now() < asset.cooldownEnd:
        return  # 冷却期内不变卦
    
    # 评估新状态
    new_hexagram, binary_state = evaluate_hexagram(asset_id)
    old_hexagram = asset.currentHexagram
    
    if new_hexagram == old_hexagram:
        return  # 无变化
    
    # 计算动爻
    old_binary = format(old_hexagram, '06b')
    new_binary = binary_state
    moving_yao = find_different_bits(old_binary, new_binary)
    
    # 执行变卦
    execute_transition(asset_id, old_hexagram, new_hexagram, moving_yao)
```

### 4.3 变卦执行流程

```
检测到变卦条件满足
    ↓
计算新卦象和动爻
    ↓
记录变卦事件：
├── 旧卦象、新卦象
├── 动爻位置
├── 触发时间
└── 触发原因（数据变化详情）
    ↓
更新索引：
├── 从旧卦象索引移除 asset_id
├── 添加到新卦象索引
└── 更新变卦索引 transitionIndex[old][new]
    ↓
更新资产状态：
├── currentHexagram = new_hexagram
├── trajectory.append(new_hexagram)
├── cooldownEnd = now() + 7days
└── lastUpdated = now()
    ↓
通知持有者（可选）
    ↓
如果六爻皆变（动爻=111111）：
└── 触发周期结束，新生命周期开始
```

### 4.4 防抖动设计

**冷却期机制：**

```
规则：
1. 同一卦象至少停留 7天
2. 7天内即使数据变化，也不变卦
3. 冷却期后重新评估

例外（立即变卦）：
- 重大事件：24小时内被引用 >50次
- 社区投票：持有者2/3同意立即变卦
- 违规检测：发现刷量等异常行为

冷却期显示：
- 用户界面显示"当前状态稳定期"
- 预计下次评估时间
- 距离变卦还需什么条件
```

## 五、动爻与周期管理

### 5.1 动爻检测

```
旧卦象：101110（泽风大过）
新卦象：001110（火风鼎）
对比：   ↑
      第6位变化（从1变0）

→ 动爻在上九（第6爻）
→ 变卦性质：上爻变化，周期尾声
→ 提示：准备进入新生命周期
```

### 5.2 动爻数量与变化强度

| 动爻数 | 变化强度 | 处理方式 |
|:------:|:--------:|:---------|
| 1爻 | 渐进 | 正常变卦，更新状态 |
| 2-3爻 | 显著 | 变卦 + 记录原因分析 |
| 4-5爻 | 剧烈 | 变卦 + 通知持有者 + 记录里程碑 |
| 6爻 | 周期结束 | 触发"用九"，新生命周期开始 |

### 5.3 用九触发（六爻皆变）

```
条件：动爻 = 111111（所有位都变）
    ↓
触发"用九·群龙无首"
    ↓
执行：
├── 当前周期历史存档（不可改）
├── 六爻重置为初九
├── 卦象重新计算（基于新周期数据）
├── 气数重置（或给予周期奖励加成）
└── 发布"新生证明"NFT

新周期特征：
- 历史轨迹可查看，但不影响新周期计算
- 持有者获得"资深"标签
- 自定价性可能提升（历史证明价值）
```

## 七、性能优化建议

### 7.1 查询优化

```
热点卦象缓存：
- 缓存访问频率最高的卦象索引
- LRU淘汰策略

异步索引更新：
- 变卦时异步更新索引
- 优先保证查询性能

分片存储：
- 按卦象ID范围分片
- 并行查询
```

### 7.2 变卦评估优化

```
事件驱动 + 定时检查：
- 实时事件触发评估（如引用发生时）
- 每日批量检查（非活跃资产）

增量计算：
- 只计算变化的数据维度
- 避免全量重新计算

预估提示：
- 距离变卦还需多少使用/引用
- 用户可主动推动变卦
```
