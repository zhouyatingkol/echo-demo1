## 第一部分：涌现物理协议（Emergent Physics）

### 核心变化对照表

| 维度 | v1.0（旧） | v2.0（新） | 影响 |
|------|-----------|-----------|------|
| **阈值** | 固定常量（7天3次） | 动态75%分位数 | 每周自动调整，由全网数据决定 |
| **六爻** | 绝对档位（达标升级） | 相对排名（竞争位置） | 同样是3次，上周算活跃，这周可能不算 |
| **验证** | "禁止违规" | "标记异常" | 不阻止，只记录。异常可能是刷量，也可能是新物理 |
| **角色** | 平台设计者 | 物理观测者 | 我们不制定规则，我们记录涌现的物理法则 |

### 关键公式

```
阈值 = percentile(上周全网数据, 75%)

六爻 = 相对排名：
  初九：后25%
  九二：25-50%
  九三：50-70%
  九四：70-85%
  九五：85-95%
  上九：前5%

异常检测：Z-Score > 3 → 标记为"值得关注的异常"
```

### 架构变化

```
v1.0: 纯链上（每次交易都计算，Gas高）
v2.0: 链下统计 + 链上验证（每周批量更新，Gas低）

链下：
  数据聚合 → 统计计算（分位数、排名） → 生成ZK proof
链上：
  验证proof → 更新全局阈值 → 记录异常事件
```

### 给创作者的叙事变化

| v1.0 | v2.0 |
|------|------|
| "遵守我们的规则" | "塑造物理常数" |
| "你需要X次使用才能升级" | "你在全网处于什么位置？" |
| "违规将被惩罚" | "异常被记录，可能是新发现" |
| 平台是裁判 | 没有裁判，只有过程 |

## 第二部分：64卦动态聚类

### 核心概念

> **64卦不是预设分类，是后验聚类结果。**

```
旧思维：设计64个盒子 → 把资产装进去
新思维：观察数据自然聚集 → 发现64个群落 → 给每个群落起个名字
```

### 算法流程（每周执行）

```
输入：所有资产的4维配置（用/扩/衍/益）
  ↓
降维：t-SNE/UMAP → 2D空间
  ↓
聚类：DBSCAN自动发现簇（或K-means固定64）
  ↓
命名：根据簇中心特征匹配传统卦名
  ↓
输出：资产→卦象映射表（Merkle Root上链）
```

### 卦象是流动的

```
本周：《墨竹》属于乾卦（全面领先型）
下周：可能属于姤卦（高开放型）

原因可能是：
1. 《墨竹》本身特征变了（更多被采样，更少被完整播放）
2. 新内容崛起，改变了聚类边界
3. 市场整体配置空间结构变了

**这不是《墨竹》变差了，是它的生态角色变了。**
```

### 与六爻的关系

```
六爻：相对排名（横向定位）
    "你在全网的使用排名是第82百分位"

64卦：配置聚类（纵向归类）
    "你的配置模式与姤卦特征相似"

关系：
    六爻是经纬度坐标
    64卦是所在城市
    两者共同定义资产状态
```

## 关键决策点（需要团队讨论）

### 决策1：阈值百分位

**选项：**
- A. 75%分位数（推荐）：前25%算活跃，较宽松
- B. 90%分位数：前10%算活跃，较严格
- C. 动态调整：根据市场热度自动选择

**问题：**
- 我们想鼓励更多人进入高阶（选A）还是保持稀缺性（选B）？
- 阈值变化太快会让用户困惑吗？

### 决策2：聚类算法

**选项：**
- A. DBSCAN（密度聚类）：
  - 优点：自动发现簇数量，可能发现新模式
  - 缺点：簇数量可能≠64，需要额外处理
  
- B. K-means：
  - 优点：固定64簇，结果稳定
  - 缺点：强制分割自然模式，可能错过新兴卦象

**问题：**
- 我们接受「本周只有58卦，下周可能有70卦」吗？
- 还是坚持「永远是64卦」，即使强制合并/分裂？

### 决策3：更新频率

**选项：**
- A. 每周更新（推荐）：平衡动态性和稳定性
- B. 每日更新：更敏感，但可能过度波动
- C. 每月更新：更稳定，但可能滞后

**问题：**
- 创作者需要多久看到自己的卦象变化？
- 变化太快会不会让「卦象身份」失去意义？

### 决策4：异常处理

**选项：**
- A. 仅标记（推荐）：记录异常，不采取行动
- B. 降级惩罚：异常资产强制降权
- C. 社区仲裁：标记后由DAO投票决定

**问题：**
- 我们如何区分「刷量攻击」和「病毒式传播」？
- 误伤创新内容的代价是什么？

### 决策5：命名保守度

**选项：**
- A. 严格匹配传统64卦（推荐）：只用《易经》已有卦名
- B. 允许新卦名：发现新模式时创造新名字（如「囚牛卦」）
- C. 纯数字标识：hex_01到hex_64，不赋予文化含义

**问题：**
- 创造新卦名是亵渎传统还是延续传统？
- 文化纯粹性和创新自由度如何平衡？

## 技术可行性要点

### 已确认可行

- **链下统计**：Python + sklearn 可以处理大规模数据
- **ZK证明**：zk-SNARKs可以证明统计计算正确性
- **Merkle Tree**：链上验证映射表的成本可控（~1000 gas/资产）
- **批量更新**：每周一次，不会阻塞网络

### 需要验证

- **冷启动**：前1000个资产之前，阈值怎么定？
- **极端情况**：如果90%资产都沉寂，阈值会失效吗？
- **数据可用性**：如果链下节点下线，历史数据怎么恢复？

**附：术语速查**

- **涌现（Emergence）**：整体性质不能从个体简单推导
- **分位数（Percentile）**：75%分位数 = 75%的数据小于此值
- **Z-Score**：标准分数，>3表示异常偏离
- **ZK Proof**：零知识证明，证明计算正确而不泄露数据
- **DBSCAN**：密度聚类，自动发现簇数量
- **t-SNE**：降维算法，保留局部结构
